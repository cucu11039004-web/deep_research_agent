# Deep Research Agent (< 500 lines)

A minimal harness-style deep research agent. Built in 6 lessons during my gap period (April-May 2026), as part of an agent engineering self-study plan.

**Blog post**: [《450 行写一个 deep research agent，然后 trace 数据吓了我一跳》](your-blog-url)

## What it does

Input: a research query in natural language.
Output: a markdown report with citations, generated by:

```
[user query]
     ↓
   [planner]              ← 1 LLM call, decomposes the query
     ↓
[sub-q1] [sub-q2] [sub-q3]   ← concurrent
   ↓       ↓       ↓
 [researcher × N]            ← each is a full harness loop
   ↓       ↓       ↓
[note1] [note2] [note3]
     ↓
   [writer]                ← 1 LLM call, synthesizes
     ↓
[Final report with citations]
```

This is the Orchestrator-Workers pattern from Anthropic's "Building Effective Agents".

## Why harness, not LangGraph

Deep research is fundamentally a model-driven loop — the planner decides what to search next based on what it just saw. Forcing this into a graph (LangGraph) adds abstraction without payoff at this scale. The whole point of this project is to **see** the bare agent loop instead of being a framework user.

The core harness is ~50 lines:

```python
async def run_researcher(sub_question, max_steps=8):
    messages = [
        {"role": "system", "content": RESEARCHER_PROMPT},
        {"role": "user", "content": f"Sub-question: {sub_question}"},
    ]
    for step in range(max_steps):
        response = await client.chat.completions.create(
            model=MODEL, messages=messages,
            tools=TOOLS_SCHEMA, tool_choice="auto",
        )
        msg = response.choices[0].message
        messages.append(msg.model_dump(exclude_none=True))
        if not msg.tool_calls:
            return msg.content  # natural exit
        for tool_call in msg.tool_calls:
            result = await dispatch_tool(tool_call.function.name, ...)
            messages.append({"role": "tool", "tool_call_id": tool_call.id, "content": result})
    # forced closure: see code for graceful degradation
```

## Findings & known limitations

See [`findings.md`](findings.md) for two bugs I discovered via trace analysis but did not fix in this sprint:

1. **Planner's "ONE aspect" rule is too weak**. It does not distinguish single-aspect questions from single-information-source questions. 80% of complex queries trigger the forced closure fallback because of this.
2. **Researcher repeats failed fetch_url calls** due to lost-in-the-middle. Same dead URL gets retried 3-5 steps later in the same run.

These are the seeds for Sprint 2 / Sprint 3.

## Design decisions worth flagging

- **DeepSeek over Claude/GPT for cost**: a 5-query test run costs <$0.10 on DeepSeek. Multi-LLM-call agents are sensitive to per-call cost.
- **Hand-written tool schemas, no `@tool` decorator**: the schema is what the model actually sees. Hiding it behind a decorator is convenient but breaks the mental model.
- **Tools return strings, even for errors**: the model needs to see error messages, not catch exceptions. `"Error: HTTP 404 for {url}"` is more useful than a Python exception.
- **`tool_choice="none"` for forced closure**: hard API constraint > soft prompt instruction. Always.
- **JSONL trace, no Langfuse yet**: trace as data is the foundation. Visualization is Sprint 4.
